Best thing to do here if you can't google the terms is look at the source for this presentation. Notes have been added for each item.
A website without logins and databases is harder to hack and and less rewarding if hacked.
Hacking a static site or flat site requires hacking the webhost, which is much more difficult. Not having scripts on the site makes it easier to remove any bad stuff that might get added. Also, you rebuild from your source with every update, so re-uploading the site is simple.

Installation

Install Ruby then 

gem install jekyll bundler

   On Windows, an encoding error may require 
chcp 65001

For simplicity in installing via Windows, I recommend Windows Subsystem for Linux. chcp command changes the codepage localization of the console. Nerd stuff.

Docker

docker pull jekyll/jekyll

If you already have docker installed, this is the easiest way of doing it besides Github Pages

Basic Use

  • Prepare
    > jekyll new sitename

  • Serve
    > jekyll s

  • Build
    > jekyll b
Contrary to what you'd imagine, a build is the de facto site where as a served site mirrors changes saved. Prepare the site, then serve it to do local tests. Build to have a production copy.

Advanced Multiuser Blogging

  1. create `_data` directory
  2. create `authors.yml` in the _data Directory
  3. populate it with author and bio info
  4. add author to post front matter
  5. optionally add bio in posts
https://blog.sorryapp.com/blogging-with-jekyll/2014/02/06/adding-authors-to-your-jekyll-site.html is a great explanation on this. Also, one could theoretically use OS level controls here to restrict posts from one user to one posts folder and force the bio to be included.
The simplicity makes it scriptable.
